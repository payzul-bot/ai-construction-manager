# AI Construction Platform — Roadmap

## 1. Назначение документа

Этот документ описывает поэтапный план развития AI Construction Platform.

Цель roadmap:
- зафиксировать приоритеты;
- синхронизировать продукт, архитектуру и бизнес;
- исключить хаотичное добавление фич;
- показать путь от MVP к масштабируемой платформе.

Roadmap — это **намерение**, а не контракт. Архитектурные принципы остаются неизменными.

> Примечание о терминологии фаз: «Фаза 1» в документе DEVELOPMENT_MODE.md — это
> локальный режим разработки (скелет и визуализация) и **предшествует** Roadmap Phase 1.

---

## 2. Принципы формирования roadmap

1. Сначала ядро, потом UX.
2. Детерминированный расчёт важнее “умных” фич.
3. Расширение идёт через данные и конфигурации.
4. Каждая фаза должна усиливать data moat.
5. Новая функциональность не ломает существующие контракты.

---

## 3. Phase 0 — Foundation (готово / почти готово)

**Цель:** зафиксировать фундамент платформы.

Состояние:
- архитектурные принципы описаны;
- доменная модель зафиксирована;
- контракт расчётного ядра определён;
- канонический словарь введён;
- базовые UX-флоу описаны;
- требования безопасности сформулированы.

Результат:
- единое понимание системы;
- готовность к системной разработке;
- отсутствие архитектурного долга на старте.

---

## 4. Phase 1 — MVP Core (0–3 месяца)

**Цель:** рабочее ядро + базовый пользовательский сценарий.

Функциональность:
- создание проекта;
- ввод данных через wizard;
- базовый Flow A (свободный текст);
- расчёт 1–3 эталонных WorkLeaf;
- формирование версии результата;
- отображение работ, ресурсов, QC и рисков.

Технические задачи:
- реализация Core Engine;
- реализация CalculationProfile;
- Rule Engine (базовый);
- версионирование расчётов;
- API для расчёта.

Результат:
- детерминированный расчёт;
- первые реальные проекты;
- начало накопления данных.

---

## 5. Phase 2 — Imports & Productivity (3–6 месяцев)

**Цель:** ускорение ввода и рост ценности для подрядчиков.

Функциональность:
- импорт Excel / CSV;
- сопоставление работ;
- подтверждение данных;
- сохранение артефактов импорта;
- перерасчёт с версиями.

Технические задачи:
- import pipeline;
- mapping UI;
- валидация импортированных данных;
- audit импорта.

Результат:
- снижение ручного ввода;
- рост вовлечённости B2B;
- расширение data moat.

---

## 6. Phase 3 — B2B & Collaboration (6–9 месяцев)

**Цель:** командная работа и корпоративное использование.

Функциональность:
- несколько пользователей в проекте;
- роли и права;
- совместное редактирование входных данных;
- история изменений;
- сравнение версий.

Технические задачи:
- RBAC;
- tenant-level policies;
- audit trail;
- уведомления о блокировках и изменениях.

Результат:
- платформа становится рабочим инструментом;
- рост switching cost;
- первые корпоративные клиенты.

---

## 7. Phase 4 — Domain Expansion (9–15 месяцев)

**Цель:** масштабирование по видам работ и объектам.

Функциональность:
- добавление новых CalculationProfile;
- расширение словаря работ;
- региональные rule packs;
- дополнительные QC и риски.

Технические задачи:
- инструменты управления конфигурациями;
- версионирование правил;
- тестирование профилей.

Результат:
- расширение рынка;
- усиление доменной экспертизы;
- рост сложности копирования.

---

## 8. Phase 5 — Procurement & Ecosystem (15–24 месяца)

**Цель:** интеграция в экосистему строительства.

Функциональность:
- сопоставление ресурсов с поставщиками;
- экспорт корзин;
- партнёрские интеграции;
- API для внешних систем.

Принцип:
- платформа не становится магазином;
- платформа остаётся инфраструктурой.

Результат:
- дополнительные источники выручки;
- ecosystem moat;
- рост LTV.

---

## 9. Phase 6 — Intelligence Layer (после 24 месяцев)

**Цель:** усиление платформы за счёт накопленных данных.

Функциональность:
- аналитика расхождений план/факт;
- выявление типовых ошибок;
- рекомендации по оптимизации;
- улучшение норм и правил.

Важно:
- AI не нарушает детерминизм;
- рекомендации не меняют расчёт без подтверждения.

Результат:
- глубокий data moat;
- конкурентное преимущество;
- доверие рынка.

---

## 10. Метрики успеха по фазам

Phase 1:
- корректность расчётов;
- воспроизводимость;
- отсутствие ручных правок.

Phase 2–3:
- время на ввод данных;
- количество версий;
- активные проекты.

Phase 4–6:
- глубина словаря;
- повторяемость сценариев;
- удержание B2B клиентов.

---

## 11. Риски roadmap

- попытка ускориться за счёт архитектуры;
- смешение расчёта и UX;
- преждевременные интеграции;
- избыточный AI.

Все риски снижаются соблюдением базовых документов.

---

## 12. Статус документа

Этот документ:
- используется для планирования;
- корректируется по фазам;
- не меняет фундаментальные принципы платформы.
